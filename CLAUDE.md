# CLAUDE.md

## Rules

- **Python env**: Always use `.venv/bin/python` (not system `python3`).
- **Commits**: No `Co-Authored-By` lines. Single-line messages (no body).
- **Dependencies**: Managed in `requirements/*.txt` (used by local dev and Docker).
- **Docs sync**: When modifying code, update CLAUDE.md and ARCHITECTURE.md to reflect changes.
- **No memory**: Never use the auto-memory system (no MEMORY.md, no memory files). All persistent context belongs in CLAUDE.md or ARCHITECTURE.md.
- **Error handling**: App should crash on unexpected errors. `try/except` only for expected, recoverable errors. Custom exceptions in `exceptions.py`.
- **No backward compat**: CRM models are owned by this project — no need for backward compatibility shims, legacy migration code, or re-export modules. Simplify freely.

## Project Overview

OpenOutreach — self-hosted LinkedIn automation for B2B lead generation. Playwright + stealth for browser automation, LinkedIn Voyager API for profile data, Django + Django Admin for CRM (models owned by this project).

## Commands

```bash
# Docker
make build / make up / make stop / make logs / make up-view

# Local dev
make setup    # install deps + browsers + migrate + bootstrap CRM
make run      # run daemon
make admin    # Django Admin at localhost:8000/admin/
make setup-account <username>  # Docker VNC login for an account (uses stored proxy_url). VNC on 5910, noVNC on 6090.
make browse-account <username> # Docker browser + proxy with saved cookies, no automation. VNC on 5911, noVNC on 6091.

# Testing
make test / make docker-test
pytest tests/api/test_voyager.py   # single file
pytest -k test_name                # single test
```

## Architecture (quick reference)

For detailed module docs, see `ARCHITECTURE.md`.

- **Entry**: `manage.py` — stock Django management. `rundaemon` command (premigrations → migrate → onboard → validate → task queue loop); `rundaemon --profile <username>` resolves to a `LinkedInAccount.username`. `manage.py` with no args defaults to `rundaemon`. `setup_account <username>` for interactive VNC login; `browse_account <username>` for browser-only VNC mode. Onboarding logic in `onboarding.py`: `OnboardConfig` (pure dataclass, `from_json()` classmethod), `missing_keys()`, `collect_from_wizard()`, single `apply()` write path that creates a `LinkedInAccount` (no Django auth User involved). Docker `start` script dispatches on `RUN_MODE` env var: `admin` (web server), `setup` (VNC login), `browse` (browser only), default (daemon).
- **Premigrations**: `linkedin/premigrations/` — numbered Python files for pre-Django filesystem changes (run before `migrate`). Tracked via `data/.premigrations` JSON file. Add new migrations as `NNNN_description.py` with a `forward(root_dir)` function.
- **State machine**: `enums.py:ProfileState` — QUALIFIED → READY_TO_CONNECT → PENDING → CONNECTED → COMPLETED / FAILED. Deal.state is a CharField with ProfileState choices (no Stage model). `ClosingReason` (COMPLETED/FAILED/DISQUALIFIED) on Deal.closing_reason. `Lead.disqualified=True` = permanent exclusion. LLM rejections = FAILED Deals with DISQUALIFIED closing reason (campaign-scoped).
- **Account model**: `LinkedInAccount` (was `LinkedInProfile`) is the first-class entity. No Django auth User dependency — `username` is a plain unique field on the account itself. Django `auth_user` is kept only for `/admin/` login. `Campaign.account` is a FK; the M2M `Campaign.users` is gone. A partial unique constraint enforces **one active campaign per account** (`Campaign.UniqueConstraint(fields=['account'], condition=Q(active=True), name='one_active_campaign_per_account')`). Activating a new campaign atomically deactivates the previous one via `POST /api/campaigns/<id>/activate/`. The daemon hot-swaps mid-loop: `_refresh_active_campaign()` runs at the top of every iteration, and on a change it cancels pending tasks for the old campaign, busts `session.campaigns` cache, and re-runs `heal_tasks()` for the new one.
- **Worker pool (dynamic account binding)**: `LinkedInAccount` has `proxy_url`, `claimed_by`, `claimed_at`, `last_heartbeat` fields. `linkedin/accounts/pool.py` exposes `claim_next_account(worker_id)` (atomic `SELECT FOR UPDATE SKIP LOCKED` via `Exists` subquery — cannot use `.distinct()` because Postgres rejects `FOR UPDATE` with `DISTINCT`), `release_account`, `heartbeat`, `is_still_eligible`. When `rundaemon` is started without `--profile`/`LINKEDIN_PROFILE`, the worker enters **pool mode**: it blocks on startup until it can claim an eligible account (active, not archived, has an active campaign). `run_daemon()` then calls `_refresh_account_binding(session, worker_id)` and `_refresh_proxy(session)` at the top of every loop iteration. If the current account stops being eligible (or some other worker stole the claim), the daemon releases, closes the browser, claims a new account, swaps it onto the session via `session.swap_account(new_account)`, and relaunches. If per-account `proxy_url` changed in the DB, the browser is torn down and relaunched with the new proxy — Playwright requires proxies at `new_context()` time. `STALE_CLAIM_TIMEOUT = 60` seconds; heartbeat cadence matches `_IDLE_POLL_INTERVAL = 5`. Graceful shutdown: `linkedin/browser/session.install_shutdown_handler()` hooks SIGTERM/SIGINT to release the claim immediately so `docker stop` doesn't leave a dead claim behind. Pinned mode (`LINKEDIN_PROFILE=<username>`) still works for debugging — pool helpers are gated on a non-empty `worker_id`. Compose: `worker-pool` service in `local.yml` omits `LINKEDIN_PROFILE` and uses empty `ports: []` so it can be scaled with `docker compose up -d --scale worker-pool=K`. Per-account proxy resolution: `linkedin/browser/login.py:_resolve_proxy_url(account)` returns `account.proxy_url` if set, otherwise falls back to env `PROXY_URL`. CRM updates a worker's live proxy via `PATCH /api/accounts/<id>/` with `{"proxy_url": "..."}` — change takes effect on the next loop iteration.
- **Task queue**: `Task` model (persistent). Five task types: `connect`, `check_pending`, `follow_up`, `send_message`, `check_inbox`. Five status values: `pending`, `running`, `completed`, `failed`, `cancelled` (cancelled is set when a task's campaign is deactivated mid-flight). Handlers in `linkedin/tasks/`, signature: `handle_*(task, session, qualifiers)`. On 401 (`AuthenticationError`), daemon calls `session.reauthenticate()` and resets the task to pending. `TaskQuerySet.pending()` sorts `send_message` tasks ahead of all other types (via a `CASE` annotation), so manual sends enqueued through `/api/messages/send/` jump the queue and run as the next task — no preemption of in-flight tasks.
- **ML pipeline**: GPR (sklearn) + BALD active learning + LLM qualification. Per-campaign models stored in `Campaign.model_blob` (DB).
- **Config**: `LLM_API_KEY`, `AI_MODEL`, `LLM_API_BASE` live in `.env` (read by `conf.py:get_llm_config()` at import time, propagated to `admin` + `worker-pool` services via `local.yml` / `production.yml`). The `SiteConfig` model still exists in the DB for legacy reasons but is no longer read by runtime code. `conf.py:CAMPAIGN_CONFIG` (timing/ML defaults), `conf.py` browser constants (`BROWSER_*`, `HUMAN_TYPE_*`), `conf.py` schedule constants (`ENABLE_ACTIVE_HOURS` flag, active hours/timezone/rest days), `conf.py` onboarding defaults (`DEFAULT_*_LIMIT`), `conf.py:FASTEMBED_CACHE_DIR` (persistent model cache, defaults to `<project>/.cache/fastembed/`), Campaign/LinkedInProfile models (Django Admin). `VOYAGER_REQUEST_TIMEOUT_MS` lives in `api/client.py` (constructor default on `PlaywrightLinkedinAPI`). `conf.py:DUMP_PAGES` (default `False`) — enable to save page HTML snapshots for fixture collection. Per-worker timing env vars: `MIN_DELAY`, `MAX_DELAY`, `ACTIVE_START_HOUR`, `ACTIVE_END_HOUR`, `ACTIVE_TIMEZONE`, `MIN_ACTION_INTERVAL`, `DEFAULT_SPREAD_WINDOW_HOURS`. REST API env vars: `API_KEY` (Bearer token for API auth), `WEBHOOK_URL` (POST target for incoming message notifications), `WEBHOOK_SECRET` (included as `X-Webhook-Secret` header), `CHECK_INBOX_INTERVAL_SECONDS` (default 300). Proxy: `PROXY_URL` env var (e.g. `http://user:pass@host:port/`). `linkedin/browser/login.py:_build_proxy_config` parses username/password out of the URL and passes them to Playwright as separate fields (Playwright does not extract embedded creds). Because the Voyager API client runs `fetch()` inside the Playwright page via `page.evaluate()`, **one proxy setting covers both browser navigation and all Voyager API traffic** — there is no separate httpx path.
- **MCP layer**: `mcp_server/` is a standalone package (no Django at runtime) that wraps every REST endpoint as a strictly-typed MCP tool. Code-mode contract: every tool takes named scalar/enum args (e.g. `send_message(campaign_id, public_id, message)`) and returns structured dicts — never accepts free-form fields like `intent` or `action`. Run with `python -m mcp_server` (stdio for Claude Desktop / Claude Code) or `--transport http` for Streamable HTTP. Env vars: `OPENOUTREACH_BASE_URL`, `OPENOUTREACH_API_KEY`, `OPENOUTREACH_TIMEOUT`. Deps in `requirements/mcp.txt` (`mcp[cli]`, `httpx`, `pydantic`). Twenty tools cover the full REST surface; see `mcp_server/server.py`. Tests in `tests/mcp/` use `httpx.MockTransport` and require no Django DB.
- **REST API**: Plain Django views in `linkedin/api_views.py`, auth via `Authorization: Bearer <API_KEY>`. Endpoints:
  - **Accounts**: `GET/POST /api/accounts/`, `GET/PATCH/DELETE /api/accounts/<id>/` (DELETE soft-archives via `is_archived`).
  - **Campaigns**: `GET/POST /api/campaigns/`, `GET/PATCH/DELETE /api/campaigns/<id>/`, `POST /api/campaigns/<id>/activate/` (atomic swap), `POST /api/campaigns/<id>/deactivate/`, `GET /api/campaigns/<id>/deals/?state=&limit=&offset=`, `GET /api/campaigns/<id>/stats/`.
  - **Leads & deals (read-only)**: `GET /api/leads/<public_id>/` (cache only — no Voyager fetch), `GET /api/leads/<public_id>/deals/`, `GET /api/deals/<id>/`.
  - **Messages**: `POST /api/messages/send/` (202 with task_id), `GET /api/messages/<public_id>/` (DB conversation history), `GET /api/tasks/<id>/` (poll status). The `send_message` handler now defaults to the Voyager `createMessage` API (single-digit-second sends); browser strategies remain as a fallback if the API call fails. API runs in the admin container; workers execute the tasks.
- **Webhooks**: `linkedin/webhooks.py:fire_webhook()` POSTs to `WEBHOOK_URL` with retry (3x exponential backoff via tenacity). Events: `message.received` (from `check_inbox` task), `message.sent`/`message.failed` (from `send_message` task). `check_inbox` tasks are seeded per campaign in `heal_tasks()` when `WEBHOOK_URL` is configured.
- **Lazy accessors**: `Lead.get_profile(session)`, `Lead.get_urn(session)`, `Lead.get_embedding(session)` — fetch from API and cache in DB on first access. Chained: `get_embedding` → `get_profile` → Voyager API. `Lead.to_profile_dict()` reads existing data only. `AccountSession.campaigns` (cached_property, list — invalidate via `session.invalidate_campaigns_cache()`). `AccountSession.self_profile` (cached_property, reads from `LinkedInAccount.self_lead`, discovers via API on first run). `AccountSession.account` is the FK reference (was `linkedin_profile` / `django_user`).
- **Django apps**: `linkedin` (main — `LinkedInAccount` + `Campaign` with `account` FK), `crm` (Lead with embedding/Deal), `chat` (ChatMessage; `owner` field is unused — left nullable for historical reasons).
- **Data dir**: `data/` holds persistent state (`db.sqlite3`, `.premigrations`). Docker users mount volumes at `/app/data`.
- **Database**: PostgreSQL by default (env vars: `POSTGRES_*`); SQLite fallback with `DB_ENGINE=django.db.backends.sqlite3`.
- **Docker**: Playwright base image, `BUILD_ENV` arg selects requirements. Multi-account: `local.yml` runs postgres + admin + N worker containers. Each worker gets `LINKEDIN_PROFILE` env var, own VNC port. `RUN_MODE`: `admin` (web server), `browse` (browser + VNC, no daemon), `setup` (interactive login), default (daemon).
- **CI/CD**: `.github/workflows/tests.yml` (pytest), `deploy.yml` (build + push to ghcr.io).
